# Main.java 설계 근거 정리

이 문서는 **BOJ 1976 여행 가자 문제**에서 **Union-Find 알고리즘을 선택한 판단 과정**을 설명합니다.

---

## 1️⃣ 알고리즘 선택 근거: Union-Find

### 문제 특성

* **무방향 그래프**: 도시 간 연결이 양방향
* **연결 요소 찾기**: 같은 연결 요소에 속한 도시들은 서로 도달 가능
* **여행 가능 여부 판별**: 여행 계획의 연속된 두 도시가 같은 연결 요소에 속해 있는지 확인
* 입력 크기: N ≤ 200, M ≤ 1,000

### 필요한 알고리즘 조건

* 그래프의 연결 요소를 효율적으로 찾을 수 있어야 함
* 두 도시가 같은 연결 요소에 속해 있는지 빠르게 확인 가능
* O(N²) 시간 복잡도
* 구현이 간단하고 직관적

➡️ 이러한 조건을 가장 잘 만족하는 방법이 **Union-Find (Disjoint Set Union)** 입니다.

---

## 2️⃣ 다른 알고리즘과의 비교

### DFS/BFS를 사용하지 않은 이유

**DFS/BFS 접근 방법:**
- 각 도시에서 DFS/BFS로 도달 가능한 모든 도시를 찾기
- 여행 계획의 연속된 두 도시가 서로 도달 가능한지 확인
- 시간 복잡도: O(N²)

**Union-Find가 더 나은 이유:**
1. **구현 간결성**: Union-Find는 코드가 더 짧고 직관적
2. **효율성**: 경로 압축으로 거의 O(1)에 가까운 Find 연산
3. **목적에 부합**: 연결 요소만 필요하므로 DFS/BFS의 경로 탐색 기능은 불필요

**결론**: 연결 요소만 확인하면 되므로 Union-Find가 더 적합합니다.

### Floyd-Warshall을 사용하지 않은 이유

**Floyd-Warshall 접근 방법:**
- 모든 도시 쌍의 연결성을 미리 계산
- 여행 계획의 연속된 두 도시가 연결되어 있는지 확인
- 시간 복잡도: O(N³)

**문제점:**
1. **과도한 복잡도**: O(N³) 시간 복잡도로 불필요하게 느림
2. **메모리 사용**: O(N²) 추가 메모리 필요
3. **단순한 문제**: 연결성만 확인하면 되므로 과한 방법

**결론**: 모든 쌍의 최단 경로가 필요하지 않으므로 Floyd-Warshall은 과도합니다.

---

## 3️⃣ Union-Find 선택의 장점

### 1. 문제 목적과의 정확한 일치

* **문제 목적**: 두 도시가 같은 연결 요소에 속해 있는지 확인
* **Union-Find 목적**: 두 원소가 같은 그룹에 속해 있는지 확인
* **완벽한 일치**: 연결 여부 = 그룹 분류

### 2. 효율적인 시간 복잡도

* **Union-Find**: O(N² × α(N)) ≈ O(N²) (경로 압축 사용)
* **DFS/BFS**: O(N²)
* **Floyd-Warshall**: O(N³)

Union-Find가 실질적으로 가장 빠릅니다.

### 3. 간단한 구현

* **Union-Find**: Find와 Union 함수만 구현하면 됨
* **DFS/BFS**: 그래프 표현, 방문 체크, 탐색 로직 등 복잡
* **Floyd-Warshall**: 3중 반복문과 복잡한 로직

Union-Find가 가장 간단합니다.

### 4. 메모리 효율성

* **Union-Find**: O(N) 공간 (부모 배열만)
* **DFS/BFS**: O(N) 공간 (방문 배열, 그래프 표현)
* **Floyd-Warshall**: O(N²) 공간 (연결성 행렬)

Union-Find가 메모리 효율적입니다.

---

## ✨ 결론

이 문제는 **그래프의 연결 요소를 찾아서 여행 계획이 가능한지 판별하는** 문제로, Union-Find 알고리즘을 사용하는 것이 가장 적합합니다.

**선택 근거 요약:**
1. 문제 목적과 완벽히 일치 (연결 여부 = 그룹 분류)
2. 가장 효율적인 시간 복잡도 (O(N²))
3. 가장 간단한 구현
4. 메모리 효율적
